# 「醒了吗?」产品需求文档

版本：V1.1  
更新时间：2026-03-30  
文档类型：当前产品口径版

## 1. 产品定义

### 1.1 产品名称

「醒了吗?」 / WakeMeUp

### 1.2 产品定位

一个低打扰、以“饮酒安全兜底 + 清醒时间估算”为核心的产品。

它不是戒酒教育工具，也不是酒量炫耀工具，而是一个：

- 平时尽量隐形
- 关键时刻有守候
- 最终要能靠原生能力提供真正兜底

### 1.3 当前产品形态

当前产品分为两层：

- Web 版：用于快速验证算法、文案、交互
- iOS 版：用于承载正式守候、通知、支付和长期产品化

### 1.4 核心目标

用户在酒局中只用极少操作，就能知道：

- 喝了多少
- 还要多久缓过来
- 是否已经超过自己的量
- 如果长时间没动静，系统会不会帮他守候

## 2. 核心用户

### 2.1 主用户

25-60 岁，存在应酬、聚餐、社交饮酒场景的成年人。

### 2.2 用户特征

- 不想在酒局里做复杂输入
- 不喜欢被教育
- 更在意“多久能醒”和“有没有兜底”
- 对体面、隐私、低打扰很敏感

## 3. 核心价值

### 3.1 即时判断

让用户快速知道当前饮酒强度与大概清醒时间。

### 3.2 静默守候

达到一定风险后，系统开始观察，并在必要时提醒或触发更强兜底。

### 3.3 低负担记录

主操作尽量只剩：

- 点 `+`
- 点 `-`
- 点“我已安全 / 结束酒局”

### 3.4 分层商业化

基础记录和估算保持低门槛；真正有价值的守候和更强兜底能力放到 Pro。

## 4. 产品原则

### 4.1 不说教

不做“戒酒打卡”“教育型折线图”“自律报告”这类重负担功能。

### 4.2 不打断酒局

喝酒时只给最少的交互和最短的决策链路。

### 4.3 体面优先

产品语言优先使用“应酬”“报个平安”“守候”而不是“酗酒”“出事”。

### 4.4 原生优先承接安全能力

凡是涉及锁屏提醒、后台守候、支付解锁、多联系人等正式能力，应优先落在 iOS 原生版本。

## 5. 功能范围

### 5.1 本机档案

当前档案字段：

- 性别
- 体重
- 身高
- 年龄
- 你的量（纯酒精阈值）
- 紧急联系人

### 5.2 酒局记录

支持四类酒：

- 啤酒
- 红酒
- 洋酒
- 白酒

每类支持：

- `+1`
- `-1`
- 首次记录时调整度数和容量

### 5.3 酒局开始确认

首次开始一局时，需要补充：

- 今天吃了吗
- 从什么时候开始喝

这是为了提高 BAC 估算精度。

### 5.4 结果展示

核心展示应包含：

- 当前累计纯酒精
- 当前风险状态
- 还要多久
- 大概几点缓过来
- 安全守候状态
- 最近提醒

### 5.5 安全守候

当前逻辑仍然保留三层：

- 到 50%：平安报备
- 超过你的量且 30 分钟无操作：预警
- 超过你的量且 60 分钟无操作：紧急提醒

但正式守候能力的长期承载端是 iOS 原生通知，而不是 Web。

### 5.6 酒局结束

用户可以手动点击：

- 我已安全 / 结束酒局

结束后应停止守候，并在原生端进入可复盘、可归档状态。

### 5.7 历史与复盘

当前方向已经明确要支持：

- 历史酒局归档
- 峰值 BAC
- 大概缓过来的时间
- 后续扩展到导出与复盘

该能力优先在 iOS 落地。

### 5.8 Pro / 支付

当前产品方向已明确存在 Free / Pro：

- Free：完整记录、基础 BAC 估算、基础界面
- Pro：守候提醒、多联系人、更强兜底、历史 / 导出 / 自定义酒类等

支付形态当前倾向：

- 一次买断终身 Pro
- 同时保留“支持者”口径作为可选品牌表达

## 6. 算法规则

当前采用 Widmark / TBW 事件流模拟，而不是简单线性除法。

### 6.1 输入

- 每次饮酒事件 `drinkEvents`
- 用户体重 / 身高 / 年龄 / 性别
- 进食状态

### 6.2 输出

三段式：

- 离清醒还差得远
- 快清醒了
- 基本清醒了

### 6.3 原则

算法用于“安全兜底估算”，不是医疗诊断结果。

## 7. 平台策略

### 7.1 Web

定位：

- 快速试错
- 演示
- 算法与文案验证

不承担最终守候能力。

### 7.2 iOS

定位：

- 正式产品主线
- 锁屏提醒
- 后续支付
- 历史复盘
- 更强安全兜底

## 8. 成功标准

当前阶段重点不是“功能越多越好”，而是：

- 用户能否低成本完成记录
- 用户能否看懂风险和清醒时间
- 守候逻辑是否可信
- iOS 原生形态是否能承接真正兜底
- 用户是否愿意为 Pro 的安全能力付费

## 9. 当前非目标

暂不追求：

- 社区
- 排行榜
- 戒酒训练
- 卡路里统计
- 重度数据分析面板
- Android 正式版

## 10. 当前最需要继续推进的方向

- 冻结 Free / Pro 边界
- 接通 iOS 支付商品
- 提升 iOS 守候质量
- 完善历史 / 导出 / 多联系人
- 继续用 Web 做轻量验证，不让它反客为主
